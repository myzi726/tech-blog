프로젝트를 진행하다보면 CSR 방식, SSR 방식 등의 용어를 흔히 접할 수 있는데, <br />
여기서 <b>CSR</b> 방식과 <b>SSR</b> 방식이 무엇인지와 어떤 차이점이 있는지를 정리해보려고 한다.

<br />

### CSR (Client-Side Rendering)

CSR은 렌더링을 전부 <b>브라우저</b>에서 처리하는 방식이다. 서버에서는 빈 HTML 뼈대와 JS 번들 파일만을 내려준다. <br />
실제 화면 구성은 JS 파일 실행 이후에 시작된다.

<b>동작 흐름</b>

1. 사용자가 웹사이트에 방문하면 브라우저가 서버에 페이지 콘텐츠를 요청한다.
2. 서버는 실제 콘텐츠가 없는 빈 뼈대의 HTML과 Javascript 파일 경로를 응답으로 전달한다.
3. 브라우저는 HTML에 연결된 Javascript 링크를 따라 서버로부터 Javascript 파일을 다시 다운로드한다.
4. 다운로드된 Javscript 파일이 실행되면서 API 요청을 통해 필요한 데이터를 가져온다.
5. Javascript는 전달받은 데이터를 기반으로 페이지를 동적으로 구성하여 브라우저 화면에 렌더링한다.

<br />

### SSR (Server-Side Rendering)

SSR은 <b>서버</b>에서 HTML을 미리 만들어서 내려주는 방식이다. 브라우저는 도착한 HTML을 바로 그리게 된다. <br />
JS 파일은 이미 만들어진 화면에 인터랙션을 붙이기 위해 존재한다.

<b>동작 흐름</b>

1. 유저가 웹사이트에 방문하면, 브라우저가 서버에 페이지 콘텐츠를 요청한다.
2. 서버는 해당 페이지에 필요한 데이터를 데이터베이스나 외부 API로부터 먼저 조회한다.
3. 조회한 데이터를 바탕으로 서버에서 HTML을 완성된 형태로 생성한다.
4. 서버는 콘텐츠가 이미 포함된 HTML을 브라우저에 응답으로 전달한다.
5. 브라우저는 전달받은 HTML을 즉시 렌더링하여 사용자에게 화면을 보여준다.
6. 이후 브라우저는 JavaScript 파일을 다운로드하고 실행하여, 페이지에 사용자 인터랙션을 추가한다. (Hydration 과정)

<br />

### <b>CSR의 장/단점</b>

장점

1. 초기 로딩 이후 페이지 이동이 빠르다.
2. 서버의 부담이 적다.
3. 인터랙션에 강하다.

단점

1. 초기 로딩이 느리다.
2. SEO에 불리할 수 있다.
   - 초기 HTML이 거의 비어있기 때문에 검색 엔진 크롤러가 콘텐츠를 제대로 수집하기 어렵다.

<br />

### <b>SSR의 장/단점</b>

장점

1. 초기 로딩이 빠르다.
2. SEO에 유리하다.

단점

1. 서버 부하
2. TTV(Time to View)와 TTI(Time to Interact) 간 시간 간격 차이

<br />

### <b>📍 표로 비교해보기</b>

| 구분           | CSR           | SSR               |
| -------------- | ------------- | ----------------- |
| HTML 생성 위치 | 브라우저      | 서버              |
| 초기 로딩 속도 | 느림          | 빠름              |
| 인터랙션 시작  | 빠름          | hydration 이후    |
| SEO 불리함     | 유리함        | 유리함            |
| 서버 부하      | 낮음          | 높음              |
| 구조 복잡도    | 비교적 단순함 | 상대적으로 복잡함 |

<br />

### <b>💡 언제 사용하면 좋을까?</b>

<b>CSR</b>

- 동적 콘텐츠가 많은 페이지
- 사용자 인터랙션이 많은 페이지
- 검색 엔진 최적화가 중요하지 않은 페이지

<b>SSR</b>

- 정적 콘텐츠가 많은 페이지
- 검색 엔진 최적화가 중요한 페이지
- 사용자 인터랙션이 적은 페이지
